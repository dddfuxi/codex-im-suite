# MCP-for-Picture Maintenance Skill

适用范围：`C:\Users\admin\Documents\New project\MCP-for-Picture`

## 角色

你在这个项目里的默认职责不是一次性修 bug，而是长期维护这个图片标注 MCP 服务，以及它和其他应用之间的连接能力。

优先级：

1. 保证 MCP 服务本身稳定可用
2. 保证与外部应用的连接顺畅，尤其是 IM 桥接、Codex 沙盒、飞书等调用链
3. 保证能力通用、接口清晰、使用方便
4. 在不破坏现有调用方的前提下持续演进

## 长期维护规则

- 默认把本项目视为“通用图片理解 / 图片标注服务”，不要把实现绑死在某一个上层应用里。
- 新增能力时，优先做成通用接口、通用参数、通用输出，不要写成只服务某个临时业务的硬编码逻辑。
- 修改接口时优先保持兼容；如果必须改行为，先保留旧入口或提供清晰迁移路径。
- 优先复用现有 HTTP API、MCP tools、配置项和目录结构，不要重复造一套平行机制。
- 任何与外部系统的联动改动，都要同时考虑“调用方式、错误反馈、输出格式、调试便利性”。

## 集成维护重点

### 1. 与 Codex / 沙盒的连接

- 目标是让上层代理可以稳定把图片交给本项目处理，并拿到可直接消费的结果。
- 优先支持本地路径、base64、URL 这三种输入方式。
- 返回结果时，优先提供结构化字段和明确输出路径，方便桥接层自动处理。
- 如果上层代理需要“生成标注图后再回传给 IM”，输出必须足够稳定，便于桥接层识别。

### 1.5 CLI 入口

- 本项目应始终保留可直接从命令行调用的图片处理入口，不强迫调用方一定通过 HTTP 或 MCP。
- CLI 应优先覆盖三类常见任务：图片描述、单主体标注、多对象标注。
- CLI 参数设计应尽量与 HTTP / MCP 对齐，避免同一能力出现三套不一致的参数语义。
- 新增能力时，如果适合人工或脚本直接调用，应同步考虑是否暴露到 CLI。

### 2. 与飞书 / IM 桥接的连接

- 重点不是只让模型“看图”，而是让用户在飞书里发图、发指令后，整条链路能闭环。
- 用户说“标注一下”“起名”“按红箭头标出来”这类请求时，要优先理解为图片标注任务。
- 如果用户已经用红箭头、红框、圈选等方式给了视觉提示，后续实现应优先遵循这些提示，不要忽略。
- 如果本项目输出的是本地生成图片，要尽量保持输出规则稳定，方便桥接层自动回发。

### 3. 能力设计原则

- “通用性”优先于“只为一个场景快修”。
- “便捷性”优先于“让调用方自己拼很多参数”。
- 标注策略优先做成“先判断图像类型，再选择后端与渲染方式”的通用路由，不要为某个垂直场景写死 prompt 或分支。
- 对常见任务应尽量提供合理默认值，例如语言、风格、输出位置。
- 错误信息要直接可排查，避免只有笼统失败提示。

### 4. 图像记忆

- 本项目支持按图像指纹保存本地记忆，记忆内容应尽量描述布局、关键区域、命名规则或用户明确给出的标注偏好。
- 记忆能力应保持调用无关：CLI、HTTP、MCP 都应能写入和读取同一份本地记忆。
- 同一张图后续再次被描述或标注时，应优先把历史记忆作为辅助约束带回模型提示，但不能无条件覆盖当前图像可见事实。
- 如果用户说“记住当前布局”“记住这张图”“以后按这个标”，默认应优先走图像记忆而不是只回复文本确认。

## 标注类需求的默认处理原则

- 用户要求标注某个图时，默认先判断是：
  - 单主体标注
  - 多对象标注
  - 图像文字描述辅助定位
- 如果用户明确说“按箭头”“按我圈的地方”“给这个起名”，应优先根据用户提示选目标。
- 如果用户指令不完整，但图中有明显人工标记，优先基于标记做保守推断。
- 如果确实无法确定标注目标，优先返回清晰说明，而不是生成误导性标注。

## 代码修改原则

- 优先改现有入口：`src/server.ts`、`src/mcp-stdio.ts`、`src/annotate.ts`、`src/translate.ts`、`src/config.ts`
- 新增配置时要有清晰默认值，并尽量不破坏旧调用方
- 新增输出约定时要考虑上游桥接程序是否容易识别
- 能通过测试或本地脚本验证的改动，尽量补验证

## Git 约束

- `input/` 和 `output/` 属于本地工作目录，不提交到 Git
- 不要把临时测试图片、生成结果图、私有 key、`.env` 提交到仓库
- 上传 Git 时，重点提交源码、文档、测试和配置模板，不提交运行产物

## 每次维护这个项目时的默认检查项

1. MCP tool 是否还能正常列出和调用
2. HTTP 接口是否还能启动和返回健康状态
3. 输出路径和返回字段是否仍然稳定
4. 与上层桥接应用的调用约定是否被破坏
5. `.gitignore` 是否继续排除了 `input/`、`output/` 和敏感文件
